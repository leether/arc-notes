# GitHub 使用规范

> 适用于阿可（Arc）的 GitHub 操作
> 目标：安全、隐私、可持续

---

## 1. 安全原则

### 1.1 Token 管理

**禁止：**
- 在代码中硬编码 token
- 在提交信息中包含 token
- 在 issue、PR 评论中暴露 token
- 使用无过期时间的 token

**必须：**
- Token 仅通过环境变量或安全密钥管理工具传递
- 定期轮换 token（建议 90 天）
- 使用最小权限原则（只授予必要的权限）
- 启用 token 的审计日志

### 1.2 仓库安全

**公开仓库：**
- 不包含任何敏感信息（密码、密钥、个人数据）
- 不包含内部系统架构细节
- 不包含未公开的产品计划

**私有仓库：**
- 仍需遵循最小权限原则
- 定期审查协作者列表
- 启用分支保护规则

### 1.3 提交安全

**检查清单：**
- [ ] 提交前检查是否包含敏感信息
- [ ] 使用 `.gitignore` 排除敏感文件
- [ ] 提交信息清晰、专业，不包含内部代号
- [ ] 大文件使用 Git LFS 或外部存储

---

## 2. 隐私保护

### 2.1 个人信息

**禁止记录：**
- 真实姓名、地址、电话
- 个人照片、身份证件
- 私人聊天记录（除非明确授权）
- 地理位置信息

**处理原则：**
- 使用化名或代号代替真实姓名
- 位置信息模糊化（城市级别而非具体地址）
- 时间戳使用相对时间或模糊时间

### 2.2 第三方信息

**引用他人内容：**
- 明确标注来源
- 遵守原内容的许可协议
- 不涉及未公开的私人对话

**关于阿奇（Achi）：**
- 可以引用其公开发表的文章
- 可以分析和评论其理论
- 不披露其未公开的系统细节

### 2.3 元数据清理

**提交前清理：**
- 检查文件元数据（作者、创建时间等）
- 移除编辑器临时文件
- 移除调试日志和临时输出

---

## 3. 内容规范

### 3.1 可以发布的内容

✅ **适合公开：**
- 对公开文章的分析和评论
- 原创的理论思考和观察
- 技术实现的最佳实践
- 学习笔记和总结

### 3.2 需要谨慎的内容

⚠️ **需要审查：**
- 涉及具体系统配置的文档
- 包含内部工作流程的描述
- 引用私人对话的内容

### 3.3 禁止发布的内容

❌ **绝对禁止：**
- API 密钥、密码、token
- 内部网络架构图
- 未公开的产品路线图
- 他人的私人信息

---

## 4. 工作流程

### 4.1 提交前检查

```bash
# 1. 检查敏感信息
grep -r "ghp_" . --include="*.md" --include="*.txt"
grep -r "sk-" . --include="*.md" --include="*.txt"

# 2. 检查个人信息
grep -r "阿泽\|真实姓名\|电话\|地址" . --include="*.md"

# 3. 检查大文件
find . -type f -size +10M
```

### 4.2 提交信息规范

格式：
```
[类型] 简短描述

详细说明（可选）
```

类型：
- `docs:` 文档更新
- `analysis:` 分析文章
- `observation:` 观察记录
- `thought:` 原创思考
- `chore:` 维护性工作

示例：
```
[analysis] Add emerath baseline analysis

Systematic review of all 10 articles on emerath.com
Focus on EMERATH v15.0 theory and recursive autopoiesis
```

### 4.3 分支策略

- `main`: 主分支，只接受经过审查的内容
- `draft/`: 草稿分支，用于未完成的内容
- 不直接推送到 main，使用 PR 流程

## 5. 应急响应

### 5.1 Token 泄露

如果 token 意外泄露：
1. 立即撤销该 token
2. 生成新的 token
3. 检查仓库历史，必要时使用 BFG 清理
4. 审查最近的操作日志

### 5.2 敏感信息提交

如果敏感信息已提交：
1. 不要惊慌，不要继续提交
2. 评估影响范围
3. 使用 `git filter-branch` 或 BFG 清理历史
4. 轮换相关凭证

### 5.3 联系方式

安全问题报告：通过安全渠道联系仓库管理员
